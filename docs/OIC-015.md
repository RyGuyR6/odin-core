# OIC-015 — Engineering Intelligence

OIC-015 gives Odin a deterministic, evidence-backed engineering analysis layer.
It extends Repository Intelligence and the OIC-014 execution loop instead of
introducing a second repository scanner or orchestration path.

## Capabilities

- Architecture summaries from indexed languages, frameworks, modules, entry
  points, and architecture categories
- Design-pattern recognition with file-level evidence and confidence
- Complexity hotspots based on file size, branches, symbols, long functions,
  and dependency pressure
- Technical-debt findings for maintenance markers, dependency cycles, and
  concentrated complexity
- Ranked, behavior-preserving refactoring recommendations
- Direct and transitive dependency impact analysis
- Change-risk scoring with explicit reasons
- Test and validation recommendations
- Deterministic reports suitable for audit, caching, and regression testing

Every finding contains a stable ID, severity, confidence, rationale,
recommendation, and evidence. Heuristic results are labeled with limitations;
the service does not claim static analysis proves runtime correctness.

## API

Repository analysis requires an authenticated principal and an existing,
completed repository scan:

```http
GET /api/repositories/{owner}/{name}/engineering-analysis
    ?path=backend/app/service.py
    &path=backend/app/models.py
    &objective=Refactor%20the%20health%20service
```

`path` is optional and repeatable. Supplying targets adds a scoped dependency
impact and risk assessment. Unknown indexed paths are rejected rather than
silently ignored.

## Autonomous execution

OIC-014 plans can use the read-only `engineering.analyze` handler:

```json
{
  "id": "analyze",
  "kind": "engineering.analyze",
  "parameters": {
    "repository": "owner/repository",
    "paths": ["backend/app/service.py"],
    "objective": "Refactor the service safely"
  }
}
```

The repository may alternatively be supplied in the execution context. The
handler result is a complete serialized engineering report that later steps can
reference through OIC-014 step-output references.

## Safety and boundaries

- Analysis is read-only and does not require an approval checkpoint.
- The canonical Repository Intelligence scan is the source of truth.
- Source reads remain confined to the indexed repository root.
- Binary and oversized files are excluded from source heuristics.
- Unknown paths and unavailable scans fail closed.
- Applying any recommendation remains subject to OIC-014 workspace approval,
  validation, mutation fencing, and recovery guarantees.

## Current limitations

- Detailed cyclomatic analysis is currently Python-specific; other languages
  use structural line, symbol, and branch heuristics.
- Pattern detection is evidence-based but intentionally conservative.
- Runtime coupling, production traffic behavior, and business semantics require
  additional evidence beyond static repository analysis.
